# 보안 가이드라인

## 1. 시크릿 · 환경 변수

| 해야 함 | 하지 말 것 |
|---------|------------|
| `.env` 로컬/Render 대시보드에만 보관 | `.env` 를 git에 커밋 |
| `JWT_SECRET` 충분히 긴 랜덤 | 기본값·예제 시크릿 프로덕션 사용 |
| Gmail **앱 비밀번호** | 계정 본 비밀번호 평문 설정 |
| 배포 후 `APP_URL` 실주소 | 메일 링크가 localhost로 나감 |

`backend/.env.example` 만 저장소에 포함.

---

## 2. 인증 · 인가

### 일반 사용자

- 비밀번호는 bcrypt 해시 저장 (컨트롤러 기존 로직)  
- JWT를 `Authorization: Bearer` 로 전달  
- `requireAuth` 가 필요한 라우트에 미들웨어 적용  

### 관리자

- 별도 로그인 (`/api/admin/login`)  
- `isAdmin` 클레임 + `requireAdmin`  
- **모든** 관리 작업 라우트에 미들웨어 (backend_28)  
- 프론트는 `adminAuthToken` 전용 헤더  

### 체크

```
새 라우트 추가 시:
□ 공개 GET 인가?
□ requireAuth 필요한가?
□ requireAdmin 필요한가?
□ 소유자 검사(ownership) 필요한가?
```

## 3. 비밀번호 재설정

| 규칙 | 설명 |
|------|------|
| 랜덤 토큰 | 추측 불가 길이 |
| DB 저장 | **SHA-256 해시만** |
| 메일 | 평문 토큰은 링크에만, 짧은 만료 |
| 사용 후 | 토큰 필드 클리어 |
| 응답 | "이메일 있으면 전송" 등 계정 열거 완화 유지 |

JWT를 재설정 URL에 넣지 않습니다 (information25 방향).

---

## 4. 차단 · 신고 · IP

- 로그인 시 차단 검사 (`checkBlocked` / Block 모델)  
- 신고 시 사용자·IP 메타 (`getClientIp`) — 악용 추적  
- 관리자 dismiss/delete 권한만 신뢰  

프론트 필터만으로 신고 콘텐츠를 숨기는 것은 **보안이 아님**. 서버 플래그·관리 삭제 기준.

---

## 5. XSS · 콘텐츠

- 사용자 입력(댓글, 문의, 제목)을 `innerHTML` 에 넣을 때 이스케이프 검토  
- 가능하면 `textContent` / 안전한 템플릿  
- 관리자 공지 HTML을 허용한다면 별도 정책 명시  

---

## 6. CORS · 오리진

- 동일 오리진(Render / :5000) 권장  
- 분리 개발 시에만 localhost 프론트 → 5000 API  
- 불필요하게 `*` CORS + credential 조합 확대 금지  

---

## 7. 관리자 페이지 노출

- 사이트 일반 네비에 관리 링크를 크게 두지 않기  
- 로그인 UI는 알아도, API는 미들웨어로 막혀 있어야 함  
- 초기 `ADMIN_INPUT_PW` 를 약한 값으로 배포하지 말 것  

---

## 8. 의존성 · 서버

- `npm install` 후 lockfile 커밋 유지  
- Express 에러 핸들러에서 스택 트레이스를 프로덕션 클라이언트에 과다 노출하지 않기  
- Free tier sleep 등 운영 이슈는 보안과 별개로 DEPLOY.md 참고  

---

## 사고 대응 (간단)

1. 의심 시 `JWT_SECRET` 교체 → 전 유저 재로그인  
2. 관리자 비밀번호 변경 · 시드 재확인  
3. 악성 게시글/댓글 관리자 삭제  
4. 필요 시 IP/닉네임 Block
